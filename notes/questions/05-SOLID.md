| Principle                               | One-line definition                                                                                |
| --------------------------------------- | -------------------------------------------------------------------------------------------------- |
| **S — Single Responsibility Principle** | A class or module should have only one reason to change.                                           |
| **O — Open/Closed Principle**           | Code should be open for extension but closed for modification.                                     |
| **L — Liskov Substitution Principle**   | A subclass should be replaceable by its parent class without breaking the behavior of the program. |
| **I — Interface Segregation Principle** | Clients should not be forced to depend on methods they do not use.                                 |
| **D — Dependency Inversion Principle**  | High-level modules should depend on abstractions, not on low-level implementation details.         |
